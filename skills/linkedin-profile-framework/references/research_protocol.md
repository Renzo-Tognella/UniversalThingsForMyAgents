# Research Protocol

Use current web research whenever the answer depends on market state, recruiter expectations, trends, keywords, or in-demand skills.

## Sources To Prefer

- official platform documentation
- official product/course pages
- current job descriptions
- reputable recruitment and career sources
- recognized technical communities
- primary sources for trend claims whenever possible

## Domain Intelligence Sweep

Before making recommendations, build a map of:

- technologies the user works with
- terms commonly used in that domain
- skills adjacent to the user's role
- recruiter keywords
- proof artifacts that signal competence
- market movement in that niche
- signals of seniority or competitiveness

This sweep must be specific to the user's work, not generic to "software engineering" in general.

## Web Research Questions

- What is currently valued for this role?
- Which keywords appear repeatedly in strong job descriptions?
- Which skills are differentiators versus hygiene?
- Which trends matter now and which are noise?
- What evidence do strong candidates show in this market?
- What should be shown in headline, About, Experience, Featured, and Skills?

## Extraction Fields

Capture:

- source title
- URL
- date when available
- main claim
- relevance to the user's case
- section of LinkedIn affected
- confidence level
- direct action to take

## Synthesis Rule

Do not report a trend unless it can be tied to at least one credible source.
Do not promote a skill unless you can explain:

- why it matters
- whether it should be learned, deepened, or just communicated better
- where it should appear on LinkedIn
- what proof should support it

## Conflict Handling

If sources disagree:

- prefer primary sources
- prefer current job-market evidence over old advice
- keep the disagreement visible in the final framework
